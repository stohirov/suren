# Correctness

This is the part of suren worth reading. The renderer is ordinary; the apparatus
that holds two independent implementations to a measured contract is not.

Written for a skeptical reader. Every number here comes from
[roadmap.md](roadmap.md), which is the primary record — the lab notebook this
document distills. Where suren does not know something, this says so.

---

## Why parity, and not goldens

The obvious way to test a renderer is a golden image: render a scene, store the
output, compare next time. suren does that too — one CPU golden per corpus entry,
43 of them today (456 KB), each gated at Δ=0. But goldens are the *secondary*
gate, and the reason is a specific failure they cannot see.

**A golden is generated by the implementation it tests.** It answers "did this
change?" — never "is this right?". If the reference renderer is wrong, the golden
records the bug as expected output, forever, and every future run agrees with it.

The primary gate is therefore **differential**: render the same scene on two
independently written backends — a pure-Go f64 CPU rasterizer and an f32 WebGPU
compute shader — and diff the raw premultiplied RGBA. Two implementations do not
share an arbitrary bug unless the bug is in something they share.

The two gates answer different questions and neither is redundant:

| Gate | Compares | Catches | Blind to |
|---|---|---|---|
| **Parity** (primary) | GPU vs CPU | one backend drifting from the other | a semantic error *both* share |
| **Golden** (secondary) | CPU vs its own stored output | the CPU reference drifting | anything the reference always got wrong |

Both blind spots are real, which is why there is a third leg:
[independent oracles](#the-third-leg-independent-oracles).

**This is not hypothetical. It happened.** See [the mesh crack](#1-the-mesh-crack--the-reference-was-the-buggy-one),
where the CPU reference was the wrong backend, a golden would have enshrined the
bug permanently, and only the differential could see it.

## Tolerance is not a knob

The contract lives in `internal/parity/contract.go` and every gate in the tree
cites it. No test spells a tolerance as a bare literal. There are exactly three
tiers.

### Δ=0 — `Identical()`

**Required** wherever both backends run the same integer or analytic path, so
there is no float pipeline to diverge. A regression here is a logic bug, never a
rounding artifact, so the gate is zero and stays zero.

Today: `many-nodes`, `clip-rect`, `blend-stack-normal`, `blend-stack-overlay`,
`fallback-gradient`, and any render after a resize.

### Δ≤1 — `Quantized()`

**The floor**, and it is quantization rather than error. The CPU composites in f64
and quantizes with `clamp8(v+0.5)`; the GPU composites in f32 and quantizes with
the same round-half-up rule, written out explicitly in `raster.wgsl`'s `quant8`.
Two pipelines computing the same real value can still land on opposite sides of a
`.5` boundary.

**Why not drive it to zero?** Because you cannot, without making one backend
reproduce the *other's arithmetic* rather than the true value — f32 everywhere, or
f64 on the GPU. If the true value is 128.5, the CPU says 129 and the GPU says 128,
**both are correct**. Forcing agreement would mean picking one backend's rounding
and calling it truth. That is a worse contract than admitting the LSB. Δ≤1 is the
honest floor, and it is a statement about 8-bit output, not about the renderer.

### Δ>1 — `Budget(tol, why)`

A **bug budget with an owner**. Admitted only where a specific operation is known
to diverge, and `Why` must name that operation. This is enforced, not conventional:

```go
func (c Config) Validate() error {
	// ...
	case ModeExact:
		if c.Tol > QuantizationFloor && c.Why == "" {
			return fmt.Errorf("tolerance %d exceeds the quantization floor (%d) without "+
				"naming the operation responsible; use Budget(tol, why)", c.Tol, QuantizationFloor)
		}
```

You cannot widen a gate without writing down which arithmetic forced it.

**The budget count is the headline.** Phase 11 tightened every gate from a blanket
`tol=2` to its measured floor. Phase 13 then *retired* the only two corpus budgets
that existed — ColorDodge (Δ≤2) and ColorBurn (Δ≤3) — by fixing the divergence
they were absorbing rather than by widening anything. Both now hold at
`Quantized()`. They were **the first tolerances in this tree retired by a fix**,
and their stated reason had also been wrong: the division never was the culprit,
it only *amplified* a backdrop the two backends had quantized differently.

Today the **corpus carries no budgets at all** — all 43 entries are `Identical()`
or `Quantized()`. Exactly two budgets survive anywhere in the tree, both on
*generated* scenes rather than hand-written ones, and both `Budget(2)`:

| Budget | Owner | Why |
|---|---|---|
| `props.assocTol` | `internal/parity/props/props.go:56` | "split render quantizes its intermediate composite to 8 bits" — compositing associativity cannot be exact: the split render rounds its intermediate to 8 bits where the whole render keeps full precision. Measured: 14/16 seeds at Δ=1, 2/16 at Δ=2. |
| `fuzz.amplifiedBlend` | `internal/parity/fuzz/spec.go:432` | An amplifying operation clears the rounding decision its 1-LSB f32-vs-f64 input difference would otherwise vanish under. Δ≤2 held flat over 531,622 executions. See [below](#the-tolerance-rule-that-took-three-tries). |

(A third `Budget(255, "measuring, not gating")` appears in
`backend/gpu/conic_test.go` — that one is a *measurement*, not a gate, and is
discussed under [the conic seam](#5-the-δ0-conic-seam-probe-that-measured-nothing).)

`contract.go`'s doc block is the authority on all of this and states the same
two, with the mechanism rather than the symptom named — the ColorDodge/ColorBurn
budgets it once listed are recorded there as **retired by a fix**, since a budget
that holds for the wrong reason is a budget waiting to break.

## Exact vs perceptual mode

**Exact** is the primary gate: max per-channel delta over premultiplied RGBA. It
is colorspace-free — it never decodes, so it cannot hide a divergence behind a
transfer function.

**Perceptual** (ΔE + SSIM over sRGB→Lab) exists only for features where exactness
is *provably* unreachable. It never replaces exact mode; it is a second gate for a
named subset, and `Perceptual()` demands a `Why` exactly as `Budget()` does.

It has three parts, because ΔE alone would not be sound:

- **ΔE over the frame composited against opaque black.** The premultiplied RGB *is*
  that composite, so this is a total function — no unpremultiply division, no
  undefined colour at α=0.
- **Alpha gated separately** by `Tol`, because compositing over black makes
  transparent-black and opaque-black indistinguishable to ΔE. That blind spot is
  asserted by test.
- **SSIM** (luma, 11×11 Gaussian σ=1.5, Wang et al.) catches structural drift a
  per-pixel metric averages away.

ΔE is **CIE76, not CIEDE2000**, with its reason recorded: CIE76 overestimates
distance for saturated colours, which is a defect when *ranking* arbitrary pairs
but not when gating two renders that should already be near-identical — there it is
only ever *stricter*, and a stricter gate cannot admit a divergence CIEDE2000 would
have caught. `srgb8ToLab` is validated against published Lindbloom sRGB→Lab D65
references, not against values read back out of the implementation.

### Perceptual mode has no customer, and that is worth saying out loud

It was built in Phase 12a for exactly one anticipated feature: mesh gradients,
which the plan predicted could not reach the floor.

**Mesh landed at Δ=1 — the exact floor.** The prediction was wrong, and the reason
is precise: the plan was imagining a **Coons** patch, whose device→parameter
inverse is a per-pixel Newton solve. Gouraud triangles invert in closed form — a
2×2 solve, one division, no iteration — so mesh sits at the floor like every other
paint.

So a whole comparison mode sits in the tree with nothing routed through it. The
honest thing is to record that rather than quietly ship a feature into it to
justify the work. It stays because Phase 17's bilinear image sampling is its first
real candidate.

## The four pillars

### 1. The corpus

`internal/corpus` — **43 named entries** (live count from `go test -run
TestParityCorpus -v`), each pairing a scene with **the tolerance it earned by
measurement**, not the one that was convenient.

Coverage: solid, gradient (linear/radial/conic/conic-seam), mesh, many-nodes,
many-segments, clip-rect, clip-path, clip-path-nested, 12 blend modes, 2
blend-stacks, 12 composite operators, 4 crossed composite×blend, 2 fallback
entries, and the fuzz regressions from `regress/*.json`.

Every entry runs as a cross-backend gate **and** as a CPU golden. The golden is
gated at `Identical()` regardless of the entry's `Tol`, because those answer
different questions: `Tol` is cross-backend (two float pipelines), while a golden
is generated *by* the deterministic CPU reference, so re-rendering must reproduce
it bit-for-bit. Gating goldens at `Tol` would let the CPU drift a full LSB
undetected.

**Goldens store premultiplied bytes through an NRGBA container, not a plain PNG.**
The obvious approach was measured first: encoding an `*image.RGBA` to PNG
unpremultiplies, decoding re-premultiplies, and the round-trip **loses Δ=1 on 13%
of channels** (34056/262144). That is the *entire* `Quantized` budget spent on the
file format before the renderer is compared — the gate would be measuring PNG.

**The crossed entries exist because of a measured hole.** Blend and composite share
one packed word (`Node.Flags`: blend in bits 0-3, operator in bits 4-7). Every
`composite-*` entry left blend at Normal and every `blend-*` entry left composite at
SrcOver, so neither family could see the two being confused. Widening the shader's
blend mask from `0xF` to `0xFF` — deliberately leaking operator bits into the mode —
**and every single-axis entry still passed**: with one axis at its zero value the
leaked bits land past the end of `blendCh`'s switch, whose default returns the
source colour, which is what Normal does. The four `composite-x-blend-*` entries are
the complete set that goes red.

### 2. Property laws

`internal/parity/props` — algebraic laws checked on *generated* scenes, not fixed
goldens. A `Law` carries its own gate, because tolerance is a property of the law,
not a global knob.

- **Affine composition** — `render(node∘(A·B)) == render((node∘A)∘B)`. Δ=0 both backends.
- **Clip idempotence** — `clip(clip(x,C),C) == clip(x,C)`. Δ=0, **pixel-aligned only** (see below).
- **Compositing associativity** — for the qualifying operators. `Budget(2)`, measured.
- **Premultiply round-trips** — `premul(unpremul(p)) == p`. Δ=0 both backends.

Laws are parameterised by a `RenderFunc`, so the suite lives in the root module and
the GPU test passes its own renderer in — `props` never imports `backend/gpu` and
the cgo quarantine holds. `CheckAll` runs each law within one backend (52 CPU
subtests); `CheckAgreement` requires both backends agree (104 GPU subtests). A law
holding independently on each backend does **not** imply the backends agree, so
that is a separate gate. Every failure prints `seed=0x…`, and the seed is the repro.

**Only two of the twelve Porter-Duff operators qualify for associativity, and the
other ten are recorded rather than skipped.** The law needs the operator to be
associative *and* transparent black to be its identity. SrcOver and DstOver pass
both. Six fail the identity condition. Clear/Dst/Src hold it vacuously. **Xor is the
near-miss worth naming:** transparent black *is* its identity and its alpha *is*
associative (`a+b−2ab` is symmetric in three arguments), but its colour is not —
cx's coefficient is `(1−ay)(1−az)` under one grouping and `1−ay−az+2ay·az` under the
other, equal only when `ay·az=0`. **Alpha associativity is not associativity.**

### 3. Differential fuzz + shrink + bisect

`internal/parity/fuzz`. A scene generator seeded by one `uint64` — the seed *is*
the repro. The oracle is CPU-vs-GPU exact delta.

**`TestFuzzSweep` runs 64 fixed seeds on every `go test`**, so the differential is a
standing gate rather than something that only runs when someone remembers `-fuzz`.

- **Shrinking:** delta debugging over the node list, then per-node stripping of
  clips/stroke/dashes/gradient/shape→bbox/transform/blend/fill-rule, to a fixpoint.
  The gate is computed **once from the original scene and held fixed** — recomputing
  it per candidate would let the shrinker re-gate its way to a "failure" the
  original never had. Fixed-gate shrinking can only under-report.
- **Bisect:** `FirstDiverging` attributes a divergence to one node. It is a **linear
  scan, deliberately, where the plan said binary search** — prefix divergence is
  *not monotone*: a later opaque node can paint over the pixels where an earlier one
  diverged, so a clean prefix at k says nothing about k−1, and a binary search would
  name a node that merely sits on a monotone boundary.
  `TestFirstDivergingFindsTheEarliestOfSeveral` pins exactly the case binary search
  gets wrong.

**A find is stored as a `Spec`, not a seed.** A seed only reproduces while the
generator is untouched, and the first edit to `gen.go` would silently retire every
past find. This is not theoretical — it happened *within* the phase that built it:
excluding two blend modes shifted every seed's scene. The `Spec` is explicit JSON,
round-tripping **bit-exactly** (verified on the render, not just the struct, since
regressions are gated at Δ=0).

Volume, per the roadmap: ~1M executions in 12c, 472k against Phase 13's tightened
gates, 531,622 against Phase 15's corrected rule, 2.3M over Phase 16's widened
space. No unexplained divergence.

### 4. Per-backend reconciliation

`TestReconcileBackends` runs the whole corpus on **every backend the host exposes**,
at each entry's earned tolerance. An absent backend **skips**, never fails.

The output states its own scope. Verbatim from a run on this machine:

```
reconcile_test.go:85: reconciled 1 backend(s): [metal]
reconcile_test.go:87: NOT reconciled here (host does not expose them): [vulkan dx12]
                      — the portability claim covers [metal] only
```

**Decision: no stored GPU goldens.** Three reasons:

- It invents a **second oracle with no authority**. If Metal records 128 and Vulkan
  says 129 while the true value is 128.5, both are correct; failing Vulkan against
  Metal's recording asserts "reproduce Metal's rounding rather than the true value"
  — the exact trade the contract rejects.
- The **CPU reference is already machine-independent**, and that is measured, not
  assumed (see [FMA](#the-honest-scope-statement)). Gating each backend against the
  CPU *on its own host* composes across hosts with no shared file and no canonical
  GPU.
- It would be **weaker than what is already gated**: a GPU golden could only be held
  at the quantization floor, while several entries are gated at Δ=0 against the CPU
  reference today.

A direct backend-vs-backend gate is no better: if both pass their CPU gate at
tolerance T they are within 2T of each other automatically (asserting 2T is a
tautology), and asserting T would claim two independent f32 drivers must agree more
tightly than either agrees with the reference, which nothing supports.

### The third leg: independent oracles

Parity compares the backends to *each other*; a golden compares the CPU to *itself*.
**Both are blind to a semantic error the two renderers share, and to a bug in the
reference.** A conic sweeping the wrong way, or a barycentric solved for the wrong
corner, would render identically on both, match its golden byte-for-byte, and pass
every corpus entry.

So each feature also gets a test that **hand-computes what it means**, asking
neither renderer:

| Oracle | Location | Asserts |
|---|---|---|
| `alphaOracle` | `raster/composite_test.go` | Each operator's output alpha derived from Porter-Duff's **coverage geometry** (αs, αb as areas of two independent subsets; each operator names which parts of the Venn diagram survive) — not from the coefficient table, which would restate it. |
| `TestConicParameterMapping` | `backend/cpu/conic_test.go` | Hand-computed angles: right=0, down=0.25 (device y grows downward, so increasing atan2 sweeps clockwise), left=0.5, up=0.75, plus `Angle` rotating the start. |
| `TestMeshAtInterpolatesBarycentrically` | `paint/mesh_test.go` | Each corner, the centroid, and all three edge midpoints. |

This is the discipline a new feature must follow. See [CONTRIBUTING](../CONTRIBUTING.md).

## The findings

These carry the doc. Each one is an argument, not trivia.

### 1. The mesh crack — the reference was the buggy one

**The clearest argument in the tree for why the reference is not the oracle.**

Two triangles sharing an edge compute that edge's weight by **different
expressions**: an edge function over the denominator in one, `1 - l0 - l1` in the
other. Exact arithmetic makes them complementary — one is zero exactly when the
other is, and the two `l >= 0` tests partition the edge. Floating point makes them
**independent roundings of the same real zero**, so both can land a hair negative
and both tests fail.

Measured on `MeshScene`'s 3×3 grid, at a pixel centre landing on a shared diagonal:

| Backend | `l` in triangle A | `l` in triangle B | Result |
|---|---|---|---|
| CPU (f64) | −3.5e-17 | −8.3e-17 | **both rejected** → painted background through the mesh: a 41-pixel hairline crack |
| GPU (f32) | rounds to +0.0 | accepted | **drew correctly** |

Δ=198.

Three things fall out of this, and each inverts an intuition:

- **More precision made it worse.** Exact zero is the only value both triangles
  accept, and f64 is *better* at not landing on it. The less precise backend was the
  correct one.
- **A CPU golden would have recorded the crack as expected output forever.** The
  golden is generated by the buggy backend. It would have been green on day one and
  every day after.
- **The GPU had no bug to find.** Only *comparing* them could see it.

**The fix is agreement, not correctness.** `paint.MeshEps` (1e-5) is in **normalized
barycentric units** — a fraction of a triangle, not a distance — so it means the same
for a 3px triangle and a 300px one. It is ~300× the f32 noise measured at that pixel
(3e-8) and widens a triangle by a hundred-thousandth of itself. The slack does not
need to pick the *right* triangle, only the **same** one on both backends; Gouraud
continuity makes the two agree along a shared edge anyway.
`TestMeshAtAcceptsAPointOnASharedEdge` pins the exact field coordinates that cracked,
at the unit level, and is injection-verified: with the eps removed it is the *only*
test that fails.

**The general lesson:** a predicate whose two branches are computed by different
expressions is exact in real arithmetic and unreliable in float.

### 2. Clip idempotence is false under AA, by design

`clip(clip(x,C),C) == clip(x,C)` is intuitively obvious and **measurably false**.

Clip coverages compose by **multiplication**, so applying the same path twice
squares an edge coverage: 0.5 → 0.25. Measured **Δ=52** on an AA circle clip, versus
**Δ=0** pixel-aligned.

This is not a rounding artifact and not something tolerance should absorb — it is
what multiplicative coverage composition *means*. Skia and Cairo behave the same.

So the law is **scoped to pixel-aligned clips**, where coverage is binary, and the AA
case is recorded here rather than hidden by a widened gate. The scoping is
load-bearing rather than decorative, and that was **verified by injection**: removing
pixel-alignment from the clip generator makes the law fail immediately.

*Open decision:* product vs `min` for stacking clip coverage. `min` would make
idempotence hold under AA, but it is a semantic change touching both backends and was
not a test-phase call.

### 3. ColorDodge/ColorBurn — an ill-conditioned oracle, not a bug

Their blend derivatives are `1/(1-cs)` and `1/cs` — **unbounded**. They multiply any
input difference without bound.

Measured on the fuzzer's own find (seed `0x737`), isolated by construction:

| scene | Δ |
|---|---|
| gradient + ColorDodge over a *blended* backdrop | **17** |
| same gradient, SrcOver instead | 1 |
| **solid** + ColorDodge over the same blended backdrop | **18** |
| gradient + ColorDodge over a *plain* backdrop | 1 |

**Both backends are individually correct in every row.** The dodge node amplifies
whatever sub-LSB difference reaches it — a 1-LSB backdrop from a previous blend, or a
gradient's source colour whose f32-vs-f64 parameter differs by well under one LSB
(invisible at Δ≤1 under SrcOver, multiplied ≈54× via `dB/dcs = cb/(1-cs)²`).

There is no bug to find and no bound to gate at. So they are **excluded from the
generator** — a differential oracle needs a well-conditioned function — with the
measurement recorded. They stay covered exactly where the oracle *does* apply: the
corpus feeds them bit-identical inputs (solid over a plain backdrop), where they now
sit at **the plain floor** since Phase 13 fixed the difference they were amplifying.

**The reframing matters more than the exclusion:** Phase 10's dodge/burn budgets held
because `BlendScene` fed them pinned inputs, not because the divergence was bounded
at 2–3. A budget that holds for the wrong reason is a budget waiting to break.

### 4. The per-node quantization mismatch a 4-node generator could never find

The real find of Phase 13, and an argument about the limits of fuzzing.

The CPU composites into an 8-bit `image.RGBA`, re-quantizing after **every node**. The
shader accumulated `fbl` in f32 across all nodes in a tile and rounded **once** at the
end. That is **a different function, not a more precise one**, and the gap grows
without bound with depth:

| stacked Overlay layers | 1 | 2 | 4 | 8 | 16 | 32 | 64 |
|---|---|---|---|---|---|---|---|
| accumulate, round once | 0 | 1 | 1 | 2 | 2 | 6 | **10** |
| round per node | 0 | **0** | **0** | **0** | **0** | **0** | **0** |

**12c's generator caps at four nodes, where the effect is a single LSB — it could
never have found this.** A million fuzz executions searching a space that does not
contain the bug. The corpus entry did, because a human reasoned about depth and wrote
`sample.BlendStack`.

`blend-stack-normal` / `blend-stack-overlay` gate it at **Δ=0 forever** (pixel-aligned
solids, so nothing but rounding can move them); reverting the shader fails them at
Δ=1/Δ=3, verified.

### 5. The Δ=0 conic seam probe that measured nothing

An adversarial probe can aim at exactly the right pixel and still measure nothing.

A conic gradient's parameter **wraps**: crossing the ray at `Angle` takes `t` from just
under 1 to just over 0, so where the first and last stops differ the paint is
**discontinuous**. Linear and radial are continuous in the pixel position, so their
f32-vs-f64 parameter difference stays sub-LSB and the floor absorbs it. A discontinuity
amplifies without bound — and unlike ColorDodge, the magnitude is set by **the stop
colours** rather than the arithmetic, so there is no derivative to bound and **no budget
can be fitted**.

**The first probe measured Δ=0.** Aiming the seam exactly through a pixel centre is
*not enough*: f32 `atan2(11,37)` rounds to the same f32 as `float32(f64 atan2(11,37))`,
so both backends compute `t=0` on the nose and agree exactly.

The hazard needs the true value inside **f32's blind spot**, not merely on the ray —
hence a 1e-9 nudge to `Angle`, below f32's ~3e-8 resolution there and far above f64's.
Then `TestConicSeamDivergesWithoutBound` measures **Δ=255 on 1 pixel of 8192**, with
*both backends individually correct*: the true parameter sits 1e-9 from the wrap and
f32 cannot resolve which side it falls on.

**The control is the point.** `TestConicSeamIsExactWhenF32CanResolveIt` keeps the
nudge=0 case in the tree, without which Δ=255 could equally mean the implementation is
simply broken — and without which the probe could silently stop probing.

The seam is bounded in **extent, not magnitude**: rare rather than mild. So the
generator emits **closed conics only** (last stop = first), which makes the paint
continuous and puts conic back where linear and radial are while still exercising the
whole atan2 path, the wrap, and the inverse transform.
`TestGeneratorEmitsOnlyClosedConicGradients` pins that **in both directions** — an
exclusion satisfied by never generating a conic at all would be vacuous (1047 conics
over 2000 seeds).

*Related vacuity bug, caught by measuring rather than review:* `randStops` draws 2–4
stops; closing the loop overwrites the last colour with the first, so a **two-stop conic
comes out a constant colour** — a solid paint wearing a gradient's name. A third of every
conic generated would have exercised no atan2 at all while passing everything. Conic now
draws 3–4 stops.

### 6. `RequestAdapterOptions.BackendType` is silently ignored

A harness that reports parity for a backend that never ran is **worse than no harness**.

The obvious way to select a GPU backend is `RequestAdapterOptions.BackendType`. It is
accepted, warned about (`unsupported, use WGPUInstanceExtras.backends`), and **silently
ignored**: asking for Vulkan on this Metal-only host returns the **Metal** adapter.

A harness built on it would have printed `vulkan: PASS` having tested Metal.

The fix is two-layered: the instance-level mask (`InstanceDescriptor{Backends:…}`) does
filter — absent backends fail to yield an adapter — and `NewDeviceOn` **additionally
verifies the adapter's own reported `BackendType` matches the request**. Defence in
depth against a library that has already answered a different question than the one it
was asked.

### The tolerance rule that took three tries

Worth recording because two shipped rules were wrong the same way.

Phase 15's generator widening found a divergence in 90 seconds: seed `0xb50`, shrunk to
three nodes — opaque background, radial-gradient rect, one SoftLight rect — diverging at
Δ=2 against a Δ≤1 gate. It contained **no Porter-Duff operator at all**. The rule gave
the budget only to scenes with *two* general blend nodes, while the budget's own stated
reason already named "the 1 LSB that f32-vs-f64 coverage and **gradient** evaluation
leave" — **the rule contradicted the budget it was selecting.**

Fixed as `general >= 2 || (general >= 1 && gradient)` — **which was also wrong, and for
the same reason: it blamed a correlate.** Each rule keyed on something *near* the cause
instead of the cause. Investigated by measurement, and every step overturned a guess:

- **Which blend modes amplify.** `max|dB/dCb|` scanned numerically: Overlay **2.0**,
  SoftLight **4.0**; Normal 0; Multiply, Screen, Darken, Lighten, **HardLight**,
  Difference, Exclusion all exactly **1.0**. HardLight is Overlay's transpose and *looks*
  like it should amplify — its `2·cs` branch only runs when `cs ≤ 0.5`, so it does not.
  Where `|dCo/dCb| ≤ 1` a 1-LSB input difference **cannot** become 2, because
  `floor(x+.5)` of two reals within 1 of each other cannot differ by 2. That is
  arithmetic, not luck.
- **The gradient was never the cause.** One Overlay node over **solids only** — which the
  shipped rule gated at `Quantized` — breaches at Δ=2 in **2 of 5955** scenes. With a
  gradient, **42 of 904**. The gradient is a ~40× risk *multiplier*, which is why it was
  present in the find and why blaming it looked right. **The shipped rule had a live
  hole**, of the same species as the one it fixed.
- **A small sample nearly hid it.** Stacked amplifying modes over solids first measured 0
  breaches in 155 scenes. Forcing paints solid to get 2999 scenes found 3 (0.1%). The
  clean claims were then re-measured at that sample size before being believed.
- **Porter-Duff amplifies independently of blending.** `porterDuff` unpremultiplies
  through `s[0]/s[3]`, and operators like SrcOut and Xor manufacture near-transparent
  backdrops — so a 1-LSB premultiplied difference gets divided by an alpha the operator
  itself drove toward zero. SrcOver's fast path never unpremultiplies at all.

**Result:** the rule is now `any amplifying blend mode || any non-SrcOver composite op`,
the gradient clause is gone, and `stackedBlend` is renamed **`amplifiedBlend`** — stacking
was never the cause, and the name kept the rule pointed at the wrong variable. The loose
gate now covers **63.6%** of generated scenes, down from the shipped 85% and **below even
the pre-Phase-15 69.5%**: the oracle is stricter than it has ever been *and* no longer has
a hole.

**The meta-lesson, which applies to the seeds themselves:** adding one draw to the
generator reshuffles every seed's scene. The seeds are **not a fixed test set**, and a
green fuzz run is never coverage of a fixed space. That is the argument for storing finds
as specs, and against ever reading a green run as proof.

## The honest scope statement

**Parity is verified on Metal (Apple silicon, M4) only.**

The reconciliation harness is done and green, and it runs the corpus on every backend the
host exposes. This host exposes exactly one adapter — `metal/integrated-gpu Apple M4`;
Vulkan, DX12 and GL enumerate **zero**.

Nothing here proves Vulkan or DX12 parity. **It proves they have never been run, and says
so out loud instead of passing quietly.** Closing that needs a non-Apple runner.

### The first suspects if a non-Metal backend diverges

Two named unknowns, both from Phase 13:

**1. FMA contraction.** Go's spec permits fusing `a*b+c`, and **it does on arm64** —
verified against `math.FMA` — which makes the CPU reference's arithmetic
architecture-dependent *in principle*. It is not observable in practice: pinning every hot
expression with explicit `float64()` conversions (`geom.Matrix.Apply`, `raster/fill.go`'s
SrcOver) reproduced **every CPU golden bit-for-bit, Δ=0** — 21 goldens and ~4M channels as
the corpus stood when Phase 13 measured it; there are 43 today.

The reason is quantitative: 8-bit output is ~2.4 decimal digits, f64 carries ~15, so the
CPU's last-bit noise sits **~13 orders of magnitude below** the rounding decision it would
have to cross. **f32 has only ~4–5 orders of headroom — which is why the GPU diverges and
the CPU cannot.** Pinning the CPU would cost FMA throughput to buy nothing.

The same latitude on the GPU is **not** safe by that argument reversed, and **WGSL offers
no way to forbid contraction**. If Vulkan or DX12 diverge past the floor, contraction in
the coverage sweep, the gradient parameter, or the blend recompose is suspect one.

**2. The `routeCol` summation order.** `raster.go` runs one running total left-to-right
across the whole row; the shader collapses everything left of a tile into a scalar backdrop
in *segment* order and seeds the tile sweep with it. **Equal in exact arithmetic, not in
floating point.** Fixing it means sweeping whole rows — the exact serialization the 2D-tiling
rewrite removed (720 → ~57,600 threads) — so it is kept deliberately and recorded at
`routeCol` as an accepted contributor to the Δ≤1 floor.

### Other known-and-recorded semantic differences below the floor

- **Gradient 16-bit rounding.** The CPU's gradient is quantized to 16 bits
  (`paint.Color.RGBA()` → `/257`), so the reference never sees a continuous gradient colour.
  Mirroring it in the shader moved **nothing** — 16-bit error sits ~130× below the 8-bit
  decision. It is an artifact of reusing Go's `color.Color` convention rather than a choice
  the reference made, and **matching an artifact costs per-pixel work to enshrine an
  accident**. Recorded at `gradColor`.
- **`atan2(0,0)` is 0 in Go and UNDEFINED in WGSL.** The exact centre pixel of a conic is
  pinned to `t=0` on both sides rather than resting on a driver's corner case. Reachable, not
  theoretical — a `Center` on a pixel centre is an ordinary thing to write. Injection-verified:
  removing the guard turns the centre pixel from `{255,0,0}` into `{41,0,214}`.

## Anti-vacuity

**A test that cannot fail is worse than no test**, because it reports coverage it does not
have. This project has nearly shipped several, and each was caught by measuring rather than
reviewing.

- **The fallback gate that measured nothing.** `sample.FallbackScene`'s first draft was a
  pixel-aligned rect with a 2-stop linear ramp — which rendered at **Δ=0 on Metal with the
  fallback off**. The gate would have been decorative. Fixed with a node that diverges by
  measurement (a radial gradient's division + a circle's AA edges), and
  `TestFallbackBuysExactness` now asserts the **off/on difference** (Δ=1 → Δ=0) rather than
  the on result.
- **The corpus that could have collapsed to one scene.** A loop-variable capture bug would
  have silently made all 12 blend entries the same scene while every gate still passed.
  `TestBlendEntriesBuildDistinctScenes` guards it, and was **verified to fail when the bug is
  deliberately injected**.
- **Generated scenes that render nothing.** Every generated scene opens with an opaque
  background, but ~0.8% still render uniformly — a node can be invisible by *correct* blending
  (Darken with a source lighter than an opaque backdrop returns the backdrop exactly, for any
  alpha). Those are **skipped, not failed**, and **the skip rate is itself gated at 3%** so
  vacuity can never quietly eat the suite. `parity.NonTrivial` lives with the contract, since
  any harness rendering a scene it did not hand-write needs it.
- **The vacuity gate that was nearly raised instead of respected.** Sampling the nine
  non-degenerate composite operators uniformly took the trivial rate from 0.8% to **4.0%**,
  past the 3% gate — not a bug: `SrcOut` over an opaque backdrop is *defined* to erase, so a
  large node wipes the frame and the differential has nothing to compare. **Raising the gate
  would have inverted its purpose.** A node keeps SrcOver at 8/10 instead (1.8% measured).
- **Every failure-path component is tested by injection**, because the renderers agree today
  and a minimizer that quietly does nothing still reports "minimized". A fake divergence with
  a known culprit proves the shrinker reaches one node, strips every feature that does not
  matter, keeps the one that does, and never modifies a passing spec.

Current trivial-scene rate: **2.3%** against the 3% gate — 21% headroom, recorded so the next
generator change starts from the real number.

## What the apparatus cost, and what it caught

Two mutation-test results are worth keeping because one of them **failed to catch** and that
was the right answer.

Phase 14's fallback, three mutations: fallback disabled → caught; fingerprint ignoring the
tile mask → caught; **CPU pass ignoring the tile mask → NOT caught.** That is not a missing
test — it is the design telling the truth. Containment is enforced at *upload* (only flagged
rects are written), so the mask is a **pure optimization**, worth 33% at 25% coverage. Since
breaking it would be silent, its contract is pinned where it actually lives, in
`backend/cpu`'s `TestTileMaskConfinesWrites`, which does catch the mutation.

And the bug review found after Phase 15, which is the sharpest reuse lesson in the tree:
`cpu.nodeBounds` padded a stroked node by the stroke's half-width, which is **not** an upper
bound — a miter join reaches `miterLimit*w/2`, four times that by default, measured 8.9px
short. **The estimate had been wrong since long before Phase 14 and was harmless**, because
its only caller (`culled`) needed a node's whole bbox off-canvas before dropping it. Then
`tileCulled` reused it to drop nodes against a handful of flagged tiles, where being 8.9px
short is enough: a miter spike reaching a flagged tile that the padded bbox misses gets culled
from the CPU patch, and the patch overwrites the GPU's correct pixels. **Δ=220 over 76
channels — corruption, not a rounding delta.**

The bound now comes from `path.Stroker.MaxExtent`, which owns the miter-limit default, and
`TestFallbackKeepsMiterSpike` reproduces the old failure exactly.

**The lesson is the reuse, not the arithmetic:** an approximation is only as safe as its
callers' tolerance for error, and Phase 14 changed that tolerance without rechecking the
approximation.
